传输层概述

传输层是承上启下的一层, 与下面的层共同构成通信部分, 同时又是面向用户和最低层. 虽然网络层以经可以将数据由一个主机传输到另一个主机, 但是网络通信的主体并不是主机, 而是主机中的网络应用进程. 通信时需要对进程进行标识

端到端服务

  • 物理连接的角度, 端到端是指不同链路上(非点对点)的连接
  • 逻辑角度, 是指下层传输透明, 对用户来说, 两端是直接相连的(对等传输原理)

点对点传输指的是物理链路直连的传输, 使用接力的方法传. 端到端的传输是虚拟链路上的传输

传输层服务

  • 面向连接的传输服务: 可靠的, 有差错控制, 拥塞控制的功能. 如 TCP
  • 无连接的传输服务: 直接向目的结点传输数据, 而不确认是否有可行的传输通道, 只提供不可靠服务

TASP 和 TPDU

TASP

传输层要在用户间建立可靠的端到端连接, 需要把用户进程相互区分. TASP 就是上层访问传输层的接口, 是传输层地址. 网络实体地址 NASP 无法区分不同的应用进程, 需要借助传输层标识.

为确保传输层地址唯一, 将传输层地址分为网络 ID, 主机 ID, 主机分配端口三部分, 前面的网络 ID 和主机 ID 就是 IP 地址, 用来指定网络实体, 是 NASP. 后面的端口是传输层的特定的属性, 用来与应用进程进行一一对应

在总共 65535 个对应的端口中, 前 1024 个端口被称为常规端口, 代表了特定的应用层服务, 不能分配给其它应用. 同一时间, 同一主机不能有两个进程使用相同的端口号. 如 80 号 TCP 分配给了 WWW(对应 HTTP)服务, 23 号 TCP 端口分配给了 Telnet 服务, 21 号分配给了 FTP 服务

TPDU

传输层数据单元 TPDU, 又被称为数据段. 包括 TPDU 头部和 有效数据(来自上层的数据). 基本结构 . 基本结构
L1 固定部分 可变部分 数据
  • L1: 1 字节, 标识 TPDU 头长度, 最大为 254
  • 固定部分: 特定协议的固定部分是固定的, 包括了目的端口和源端口等. 不同的协议是不一样的
  • 可变部分: 包含一些可选参数, 每个参数有三个部分
    • Code: 1 字节, 表示类型
    • Length: 1 字节, 长度
    • Value: 值

  • 常见的 TPDU

    • CR(Connection Request): 连接请求
    • CC(Connection Confirm): 连接确认
    • DR(Disconnect Request): 断开连接请求
    • DC(Disconnect Confirm): 断开连接确认
    • DT(Data): 数据
    • AK(Acknowledgement): 数据确认
    • ED(Expedited Data): 加速数据
    • EA(Expedited data Acknowledgement): 加速数据确认
    • RJ(Reject): 拒绝
    • ER(Error): 错误

传输层服务

面向连接的传输服务需要完成 8 个服务

  1. 传输层寻址
  2. 传输层连接建立
  3. 数据传输
  4. 传输连接释放
  5. 流量控制
  6. 拥塞控制
  7. 多路复用和解复用
  8. 崩溃恢复

传输层寻址方案

应用进程可以将自己关联到一个端口上, 其中一些常见应用(如 WWW, FTP, POP, SMTP, DNS, DHCP 等)是固定分配了端口的. 而其它服务需要确认端口

进程服务器的 TSAP 分配方案

是一种"初始连接协议", 由进程服务器为各应用进程集中建立初始传输连接, 再转给对应的服务进行正式的传输连接. 进程服务器可以监听一组端口

  1. 用户向服务器发送传输连接请求, 如果没有专门的服务再等待, 用户会得到与"进程服务器" 的连接
  2. 进程服务器接收到用户的的请求后, 装载对应进程, 启动与该进程对应的服务
  3. 进程限务器为该进程分配临时端口, 用户与对应服务建立初始连接, 允许服务进程继承用户和进程服务器已有的连接, 进程服务器则继续监听新的请求

名称服务器 TASP 分配方案

一些知名服务如果用进程服务器进行分配的话, 会导致其繁忙. 而并不是所有的知名服务都是繁忙的, 仍然可以使用代理进行集中管理

为了解决以上问题, 一种新的方案 “名称服务器”, 有时也称为 “目录服务器”,

传输连接与建立

正常情况

  1. 客户端用户进程调用传输层 T_CONNEC.request 服务原语. 由所使用的传输层协议发送 CR TPDU 到服务端主机 B
  2. 主机 B 收到主机 A 的连接请求后, 调用 T_CONNECT.indication 原语, 提示对应服务进程: 有用户想要与你建立传输连接
  3. 如果服务进程接受连接请求, 则主机 B 用户实体调用 T_CONNECT.response, 然后通过传输层协议向主机 A 发送 CC TPDU
  4. 主机 A 收到 CC TPDU 后, 调用 T_CONNECT.confirm 向源用户进程进行连接确认

传输连接拒绝

服务端发到请求后因为源端服务质量差或本服务忙, 发送 RJ TPDU 拒绝连接

重复传输连接的解决办法

网络拥塞造成多次连接或因子网分组存储造成的重复连接等问题, 可以有一些解决办法

非重复 TSAP

为每次连接赋予一个新的端口, 当连接被释放时将此端口废弃. 不适用于端口固定的知名服务

过时连接表

为每个连接分配标识符, 当连接被释放时将此连接信息存入过时信息表, 如果新来的连接是过时的, 说明是重复连接. 需要唯护一张过时表, 且不适用于突变状态, 如崩溃, 重启等

分组的 TTL 机制

分组的 TTL 机制先定义每个分组的生存时间 TTL, 如果因超时未达则抛弃该分组

三次握手机制

  • 第一次握手: 发送端发送 CR TPDU, 请求建立连接
  • 第二次握手: 接收端收到 CR TPDU 后, 返回 CC TPDU, 需要在应达中宣告初始化序列号
  • 第三次握手: 发送端收到应答分组后, 发送第一个 DT TPDU

发生重复连接时, 发送端意识到应答分组中的源参考为接收端的连接已经确认, 便认为该应答是过时的

数据传输

普通数据传输, 加速数据传输

连接释放

  • 对称释放: 各自独立发送出释放连接请求, 并收到确认之后才释放连接
  • 非对称释放: 发送释放请求后单方面终止连接

流量控制

使用信号量(CDT)分配的窗口机制. 和用 CR CC AK 中的 CDT 字段, 接收端利用该字段来传递可用缓冲区大小, 也就是窗口大小

多路复用

  • 向上多路复用, 多个 TSAP 对应一个 NSAP
  • 向下多路复用, 一个 TSAP 对应多个 NSAP

崩溃恢复

第 N 层崩溃只能由保留了足够状态信息的第 N+1 层(更高一层)来完成, 对于传输层崩溃需要应用层来完成
