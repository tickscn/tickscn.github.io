结构型模型涉及到如何组合类和对象以获得更大的结构. 结构型类模式采用继承机制组合接口或实现. 结构型对象模型不是对接口和实现的组合, 而是对对象的组合

适配器模式(Adapter)

  1. 意图: 将一个类的接口转为客户希望的另一个接口
  2. 别名: 包装器(Wrapper)
  3. 动机: 有时为复用设计的工具箱类无法在该专业应用领域被复用, 仅仅是因为它的接口与所需接口不匹配
  4. 适用性
    • 想要使用一个以存在的类, 但是它的接口不符合需求
    • 想创建一个可以复用的类, 可以与不相关的类或不可预见的类协同工作
    • 想要使用一些以存在的子类, 但不能对每一个都相应的子类化以匹配接口
  5. 参与者
    • Target: 本应用使用的相关的接口
    • Client: 与符合 Target 的对象协同
    • Adaptee: 以存在的类接口
    • Adapter: 进行适配的类
  6. 效果:
    • 类适配器
      • 可以用 Adapter 修改部分 Adaptee 的实现, 因为它是 Adaptee 的子类
      • 不需要额外指针来间接获取 Adaptee 的对象
      • 当想匹配一个类和它的所有子类时, 这是不可能的
    • 对象适配器
      • 可以一次适配 Adaptee 和它的所有子类
      • 也可一次性为所有的 Adaptee 添加功能
      • 重定义实现比教困难
  7. 实现:
    • 使用 C++ 实现适配器类, 在使用 C++ 实现实现适配器类时, 公有继承 Target, 私有继承 Adaptee
    • 可插入的适配器类
      • 首先找到一个 “窄” 接口, 因为小的接口集更容易适配
      • 使用抽象接口
      • 使用代理对象

桥接模式(Bridge)

  1. 意图: 分离接口与实现, 使两者可以独立变化
  2. 别名: Handle/Body
  3. 动机: 当一个抽象可以有多个实现时, 通常用继承来协调它们. 抽象类定义接口, 不同的子类提供不同的实现. 但是不够灵活. 抽象与实现相互绑定, 无法实现部分的独立修改. 例子, 一个支持 X Window 和 PM 系统中都可使用程序, 可以使用继承的方式实现 Window 的两个子类, XWindow 和 PMWindow. 由它们来分别实现对象平台的界面. 但是有一些不足
    • 扩展 Window 使它支持不同的类型或扩展到新的窗口系统中很不方便. 如果要支持一个用来处理图标的 IconWindow. 两个窗口系统都要各实现一个对应子类, 扩展到新的窗口系统更是需要全部对应 Window 子类
    • 继承使得客户代码与平台相关, 客户创建窗口时应不涉及到其具体实现.
    • 使用桥接可以解决这些问题. 使用两个独立的类层次, Window 和 WindowpImp, 对 Window 子类的操作由对应的 Imp 子类来完成
  4. 适用性
    • 希望抽象与实现之间没有固定的绑定关系, 比如希望在运行时更换实现时
    • 对一个抽象的实现部分代码进行修改不应该影响客户
    • 在 C++ 中想要完全隐藏抽象的实现部分
  5. 参与者
    • Abstraction: 定义抽象类接口, 并维护一个指向实现类的指针
    • RefinedAbstraction: 扩充抽象类的接口
    • Implementor: 定义实现类的接口, 可以是基础操作, 由抽象类使用这些基本操作实现更高层的操作
    • ContreteImplementor: 实现的具体子类

组合模式(COMPOSITE)

  1. 意图: 将对象组合成树形结构以表示 “部分-整体” 的层次结构. Composite 使得用户对单个对象和组合对象的使用具有一致性
  2. 适用性:
    1. 表示对象的部分-整体层次结构
    2. 用户应忽略单个对象与组合对象的不同
  3. 实现
    1. 显示的父部件引用
    2. 共享组件: 一个可行的方法是为子部件存多个父部件
    3. 最大化 Component 接口
    4. 声明管理子部件的操作: 添加和删除子部件的操作放在 Component 时, 用户有可能执行向 Leaf 添加和删除组件的操作, 这是没有意义的. 而将实现放在 Composite 时, 组件与组合的接口又有不同. 一个折中的办法是 Component 中实现一个 GetComposite 操作, 默认返回空指针. 由 Composite 重写返回 this

装饰模式(Decorator)

  1. 意图: 比子类更为灵活的为一个对象添加功能
  2. 适用性:
    • 在不影响其它对象的情况下, 以动态, 透明的方式给单个 对象 添加职责
    • 处理可撤消的职责
  3. 实现
    • 接口的一致性
    • 省略抽象的 Decorator 类

外观模式(Facade)

  1. 意图: 为子系统的一组接口提供一个一致的界面
  2. 适用性:
    • 为复杂系统定制一个简单视图, 而需要更底层定制的也可以越过该层
    • 客户程序与抽象类的实现有很大的依赖, 可以引入 facade 将它们分离
    • 构建一个层次系统时, 使用 facade 模式定义每层的入口点, 让它们只通过 facade 进行通迅, 从而简化了它们的依赖关系
  3. 参与者
    • Facade: 知道子系统, 将客户主请求发给适当的子系统对象
    • Subsystem Class: 实现子系统功能, 完成 Facade 指派的任务, 并不保存 Facade 的相关信息
  4. 实现
    • 用抽象类定义 Facade 而它和子类对应不同的子系统实现
    • 公共子系统类和私有子系统类

享元模式(Flyweight)

  1. 意图: 运用共享来支持大量细粒度的对象
  2. 适用性:
    • 一个应用程序使用了大量对象
    • 由于使用了大量对象造成了很大的开销
    • 对象的大多数状态可变为外部状态
    • 如果删除对象的外部状态可以节省开销
    • 应用程序不依赖对象标识
  3. 实现:
    1. 删除外部状态
    2. 管理共享对象: 使用 Flyweight Factory 间接获取对象实例

代理模式(Proxy)

  1. 意图: 为其它模式提供代理, 控制对这个对象的访问
  2. 别名: Surrogate
  3. 适用性:
    • 在需要使用比较通用和复杂指针代替简单指针的时候
      1. 远程代理: 为一个对象在不同的局部空间提供代表
      2. 虚代理: 根据需要创建开销很大的对象
      3. 保护代理: 控制对原始对象的访问
      4. 智能指引
        • 当该对象没有被引用时, 释放它
        • 当第一次引用一个持久对象时, 将它加入内存
        • 当访问一个对象时, 检查是否己经加锁
  4. 参与者:
    • Proxy:
      • 保存一个引用(指针)使得代理可以访问实体
      • 提供一个与被代理者相同的接口, 可以用来代替
      • 控制对实体的存取, 可能还要负责创建和删除实体
    • Subjec: 定义 Proxy 和 RealSubject 的共同接口中, 这样就可以使用 Proxy 来代替 RealSubject
    • RealSubject: 被代理的实体
  5. 实现:
    • 有一些语言特性对代理模式的实现是很方便的
      • C++ 可以重载 * 与 -> 运算符

结构型模式的对比

适配器与桥接

Adapter 和 Bridge 都给另一个对象提供了一定的间接性. 而它们的目的是不同的, Adapter 是不需要重新设计己有的类就可以复用它. 而 Bridge 是对抽象接口与可能的实现进行桥接. 所以 Adapter 一般用于避免代码重复, 而 Bridge 是为了避免接口与实现的耦合

组合, 装饰和代理

装饰和组合都是基于递归组合来组织可变数目的对象, 但是组合是为了能够统一的对待, 多重对象可以被当成一个对象来处理. 而装饰是为了增加新的功能

虽然装饰和代理都保存了一个对象的引用(指针), 并向引用的对象传递请求. 两者都提供一致的接口, 但代理不能动态的的添加和分离性质
