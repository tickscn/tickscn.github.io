多线程与析构函数

当一个对象可以被多个线程访问, 如何确认何时销毁对象, 常见的竞态条件有

  • 在即将析构一个对象时, 如何确认是否有其它线程正在使用该对象的成员函数
  • 如何保证在使用成员函数时成员不会被另一个线程析构
  • 在调用某个对象的成员函数之前如何确保它还活着, 会不会它的析构函数已经执行了一半

为了解决这些问题, 可以使用智能指针

线程安全

一个对象是线程安全的是指

  • 多个线程同时访问时, 其表现出正确的行为
  • 这些线程如何被调度不影像对象的行为正确性
  • 调用端代码不需额外的同步或协调操作

MutexLock 和 MutexLockGuard

约定两个类

  • MutexLock 是指封装了临界区的资源类, 用来管理互斥器的创建和销毁
    • Windows 中是 struct CRITICAL_SECTION
    • POSIX 中是 pthread_mutex_t
  • MutexLockGuard 用来管理互斥器的加锁和解锁, 一般是个栈上的局部变量, 构造时加锁, 析构时释放. 生命周期与临界区一致

class Counter {
public:
    Counter() : _value(0) {}
    int64_t value() const
    {
        MutexLockGuard lock(_mutex);
        return _value;
    }
    int64_t getAndIncrease()
    {
        MutexLockGuard lock(_mutex);
        int64_t ret = _value++;
        return ret;
    }

private:
    Counter(Counter&);
    Counter(Counter&&);
    Counter& operator=(Counter&);
    int64_t _value;
    mutable MutexLock _mutex;
};

实际项目中使用原子量可能更合适, 这里只是一个用来表示线程安全的例子. _mutex 是各对象独有的, 不同的对象之间没有锁争用. 如果将其换为 static, 不同的对象之间共用一个锁, 不会影响正确性, 但会严重影响性能

线程安全的对象创建

要求不能泄露 this 指针.

  • 不再构造函数中汪册回调
  • 不再构造中将 this 传给跨线程的对象

为了线程安全, 有时会违背 RAII, 使用二段式构造, 使用构造函数和初始化函数来实现. 构造函数的"回头率"为 0, 作到线程安全还是比教简单的

线程安全的析构

单线程中析构只需注意避免空悬指针和野指针. 多线程中一般成员函数保证临界区不会有多个线程同时执行(一个函数如果要锁信同类型的多个对象, 需要保证加锁的顺序相同, 不然有死锁的风险. 一种方法是比效互斥量的地址, 小的先加锁), 这需要互斥量是有效的, 而析构函数会锁毁互斥量.

作为成员就量的 _mutex 无法保护析构

Raw Pointer 的缺陷

需要手动释放, 关键是无法确认该指针是否合法. 在面向对象的程序设计中, 对象的关系主要有三种, 组合(Composite), 关联(Association), 聚合(Aggregation). 组合关系中对象的生命周期由它的所有者控制, 当所有者析构时会同时把这些对象也析构关联表式对象 A 可以调用对象 B 的函数, 两者可以有不同的类型. 代码表现上看是 A 持有 B 的指针或引用. 聚合在代码表现上与关联相似, 只不过聚合的两者有逻辑上的部分-整体关系. A 可以使用 B, 但 B 的生命周期不被 A 控制. 一个可能的方法是只创建不销毁, 类似于线程池, 使用一个对象池来暂存用过的对象, 下次申请新对象时可以先查询对象池中的存货, 如果有就重复利用, 没有时再新建一个. 对象用完不是释放而是放回对象池. 这样可以避免访问失效对象. 这种方法的主要问题有

  • 如何线程安全的将对象放入对象池, 不会出现部分放回的竟态
  • 对象全局共享引发的锁争用, 有可能会将多并发操作串行化
  • 有多种类型的对象时, 使重复使用对象池还是使用类模版

一个观察者和被观察者, 被观察者通知观察者时如何确认观察者存活, 如果采用观察者析构时解注册的方法, 又有如何确认注册地还存在的问题. 即使注册地是一个长期变量也不安全. 可能出现析构还没完成, 还没有成功解注册, 就被访问的问题. 希望有一个活着的对象可以提供类似于 isAlive() 的方法来提供对象是否存活的信息. 引入一个中间对象 proxy, 指针指向 proxy, 当其中一个线程中的指针释放对象时, 由 proxy 释放并将自己的值设为 0. 其它指针(包括其它线程的)可以通过 proxy 知道目标以经死亡. 为了使 proxy 可以在适当的时候释放, 加上引用计数, 当没有引用的时候, 可以启动垃圾回收.

上面的方法就是引用计数型智能指针.

shared_ptr 和 weak_ptr

shared_ptr 是强引用, 对象的生命周期由它管理, 而 weak_ptr 并不管理对象的生命周期, 但它可以判断对象是否是一个合法对象, 如果是合法的对象, 它可以提升为 shared_ptr. C++ 中常见的内存问题有

  1. 缓冲区溢出
  2. 空悬指针/野指针
  3. 重复释放
  4. 内存泄露
  5. 不配对的 new[] 的 delete
  6. 内存碎片

前面 5 个问题可以使用智能指针解决

class Observable {
public:
    void register(const std::weak_ptr<Observer> x);
    void notifyObservers();

private:
    mutable MutexLock _mutex;
    std::vector<std::weak_ptr<Observer>> _observer;
    using Iterator = std::vector<std::weak_ptr<Observer>>::iterator;
};

void Observable::notifyObservers()
{
    MutexLockGurad lock(_mutex);
    Iterator it = _observer.begin();
    while (it != _observer.end())
    {
        shared_ptr<Observer> obj(it->lock());
        if (obj)
        {
            obj->update();
            ++it;
        }
        else
        {
            _observer.erase(it);
        }
    }
}

这解决了解注册时的竞争问题, 但有了新的问题

  1. 侵入性, 要求 Observer 必须用 shared_ptr 管理
  2. Observer 需要有 Observable 的指针(用来析构时 unregister), 为了安全就需要 Observer 用 shared_ptr 来管理
  3. 死锁. 如果 _mutex 是不可重入的, update 中如果调用了 (un)register, 会造成死锁

MutexLock mutex;
shared_ptr<Foo> globalPtr;
void doit(const shared_ptr<Foo>&);

void Read()
{
    shared_ptr<Foo> localPtr;
    {
        MutexLockGuard lock(mutex);
        localPtr = globalPtr;
    }
    doit(localPtr);
}

void Write()
{
    shared_ptr<Foo> newFoo(new Foo);
    shared_ptr<Foo> oldFoo;
    {
        MutexLockGuard lock(mutex);
        globalPtr.swap(oldFoo);
        globalPtr = newFoo;
    }
    doit(newFoo);
}

shared_ptr 技术与陷阱

  • 意外延长对象的生命周期: 因为 shared_ptr 是强引用的, 意外拷贝的话会使其管理的对象生命不合理的延长. 注意只是对象认识目的应使用 weak_ptr. 小心使用 std::bind
  • 作为参数: shared_ptr 拷贝时通常需要加锁, 为了效率在多数情况下使用常引用的方式传递参数. 只需保证在最外层函数中有一个实体对象
  • 创建时指定析构动作: 这样作的好处有
    • 虚析构不再是必须的
    • shared_ptr 可以持有任何对象, 并安全的释放
    • shared_ptr 对象可以安全的跨越模块
    • 二进制兼容性. 即使管理对象的大小变了, 只要 Foo 的头文件中不出现访问对象成员的内联函数. 并且其对象由动态库中的 Factory 创造, 返回其共享指针
    • 析构动作可以定制
  • 将析构转移到一个专门的线程

class noncopyable {

    void operator=(const noncopyable&) = delete;

protected:
    noncopyable()  = default;
    ~noncopyable() = default;
};

class StockFactory : noncopyable {
public:
    std::shared_ptr<Stock> get(const std::string& key)
    {
        shared_ptr<Stock> pStock;
        MutexLockGuard lock(_mutex);
        if (_stocks.count(key) == 0)
        {
            pStock       = make_shared<Stock>(key);
            _stocks[key] = pStock;
        }
        return pStock;
    }

private:
    mutable MutexLock _mutex;
    std::map<std::string, std::shared_ptr<Stock>> _stocks;
};

class StockFactory : noncopyable {
public:
    std::shared_ptr<Stock> get(const std::string& key)
    {
        shared_ptr<Stock> pStock;
        MutexLockGuard lock(_mutex);
        weak_ptr<Stock>& wkSocket = _stocks[key];

        pStock = wkSocket.lock();
        if (!pStock)
        {
            pStock   = make_shared<Stock>(key);
            wkSocket = pStock;
        }
        return pStock;
    }

private:
    mutable MutexLock _mutex;
    std::map<std::string, std::weak_ptr<Stock>> _stocks;
};

上面的例子还有一个小问题, Stock 可以被析构了, weak_ptr 却不行, _stocks 只增不减. 虽然这些弱引用对象都还可以访问, 但它们以是无用的对象了. 可以使用 shared_ptr 定制析构行为的功能. 在析构时调用该工厂的 deleteStock 方法删除没用的 weak_ptr. 但是为了调用该对象的成员函数, 需要传入一个该对象的 this 指针, 有可能在对象以析构时调用对象的指针造成 core dump. 所以有了弱回调技术.

    class StockFactory : public std::enable_shared_from_this, noncopyable {
public:
    std::shared_ptr<Stock> get(const std::string& key)
    {
        shared_ptr<Stock> pStock;
        MutexLock lock(_mutex);
        std::weak_ptr<Stock>& wkStock = _stock[key];
        pStock                        = wkStock.lock();
        if (!pStock)
        {
            pStock.reset(new Stock(key), std::bind(StockFactory::deleteStock, shared_from_this(), _1));
            wkStock = pStock;
        }
        return pStock;
    }

private:
    MutexLock _mutex;
    std::map < std::string, std::weak_ptr<Stock> _stock;
    void deleteStock(Stock* stock)
    {
        if (stock)
        {
            MutexLockGuard lock(_mutex);
            _stock.erase(stock->key);
        }
        delete stock;
    }
};

    class StockFactory : public std::enable_shared_from_this, noncopyable {
public:
    std::shared_ptr<Stock> get(const std::string& key)
    {
        shared_ptr<Stock> pStock;
        MutexLock lock(_mutex);
        std::weak_ptr<Stock>& wkStock = _stock[key];
        pStock                        = wkStock.lock();
        if (!pStock)
        {
            pStock.reset(new Stock(key), std::bind(StockFactory::weakDeleteCallback, std::weak_ptr<Stock>(shared_from_this()), _1));
            wkStock = pStock;
        }
        return pStock;
    }

private:
    static void weakDeleteCallback(const std::weak_ptr<StockFactory>* wkFactory, Stock* stock)
    {
        std::shared_ptr<Stock> factory(wkFactory.lock());
        if (factory)
        {
            factory->removeStock(stock);
        }
        delete stock;
    }
    void removeStock(Stock* stock)
    {
        if (stock)
        {
            MutexLockGuard lock(_mutex);
            _stock.erase(stock->key);
        }
    }

private:
    MutexLock _mutex;
    std::map < std::string, std::weak_ptr<Stock> _stock;
};
