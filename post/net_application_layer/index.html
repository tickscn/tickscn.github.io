一般网络应用都是客户端/服务器(Client/Server, C/S)模式, 主要介绍 HTTP, E-mail, DNS, DHCP

应用层概述

应用层组件和典型应用服务

应用层需要确认相互通信进程的有效性并提供同步, 需要提供应用程序或进程所需要的信息交换和远程操作, 需要错误恢复机制等通信支持服务. 这些服务统称为应用服务组件(Applicantion Service Element, ASE)

不同的组件可以让各种网络应用方便的与下层进行通信. 最重要的 3 个 ASE 是

  • 关联控制服务组件(Association Control Service Element. ACSE): 建立, 维护, 终止连接
  • 远程操作服务组件(Remote Operation Service Element, ROSE): 使用请求应答机制实现远程操作
  • 可靠传输组件(Reliable Transfer Service Element, RTSE): 将数据转为 8bit 字节串, 并将数据串分割为多个段

TCP/IP 体系的应用层

  • 支撑协议
    • 域名服务协议(DNS)
    • 动态主机分配协议(DHCP)
    • 简单网络管理协议(SNMP)
  • 应用服务
    • Web 浏览服务
    • E-mail 服务
    • 文件传输, 访问服务
    • 远程登录服务
  • 应用服务对应的协议
    • 超文本传输协议(HTTP)
    • 简单邮件传输协议(SMTP)
    • 文件传输协议(FTP)
    • 简单文件传输协议(TFTP)
    • 安全文件传输协议(SFTP)
    • 远程登录协议(Telnet)

C/S 服务模型

一方通常需要处于守候状态, 等待连接请求. 这就是 C/S 模型. 一台主机通常可以运行多个应用服务程序, 每个服务程序又有多个线程可以并发的处理多个客户的请求. 服务器上和服务程序启动后注册 TCP 或 UDP 端口. 客户端可通过与 TCP 端口建立连接或向 UDP 端口发送信息来请求服务

服务器应有对请求的并发处理能力. 可以有以下几个方案

  1. 迭代服务器(iterative server)方案: 服务器维护 FIFO 请求队列, 服务器一个一个的处理队列中的请求
  2. 并发服务器(concurrent server): 主进程只是一个等待请求和守护进程, 当有请求时, 创建一个新的线程来处理请求

Web 服务基础

目前应用中普级度最高的, 频次最多的 Web 服务是万维网(World Wide Web, WWW), 其核心协议是超文本传输协议(Hypertext Transfer Protocol, HTTP)

Web 服务模型

任何网络应用都应有一个实现服务请求, 服务提供, 有的还包括服务注册的完整系统结构, 即服务模型

从用户角度看所有的网站其实是一个巨大的文本库. Web 服务的服务模型就有三个主要部分

  1. Web 服务器(Web 服务提供者)
  2. Web 客户端(Web 服务请求者)
  3. Web 注册中心: 只有互联网中的 Web 服务需要: 负责网站的注册和管理, 并向用户提供域名解析服务

万维网的全球统一标识

在 WWW 中存在 3 种维一标识

  1. 统一资源定位器(Uniform Resource Locator, URL): 用来标识路径
  2. 统一资源标识符(Uniform Resource Identifier, URI): 用来标识资源, 可以使用路径也可以使用名称
  3. 统一资源名称(Uniform Resource Name, URN): 用来给资源命名

URL

是 URI 的一种, 其一般格式为

简单的
<协议>://<主机名>:<端口>/<路径>

  • 协议

    访问对象使用的协议, 包括 HTTP, FTP, Gopher, Telnet, File, Mailto 等

  • 主机名

    Web 服务器名称, 互联网上的主机名就是网站的域名

  • 端口

    用来指定建立连接时所用的传输层端口号, 对于常规协议, 省略端口号时会使有默认端口.

  • 路径

    用来指定要访问的对象在服务器上的文件路径

URI 和 URN

URL 和 URN 都可看为 URI 的子集. 不过 URN 的使用很少. 如国际书号 ISBN, RFC 文档, 电子邮箱账户

万维网文档标记

为了在不同的地址都能正常显示网页内容, 网站信息使用全球统一的标记方法, 超文本标记格式(Hypertext Markup Language, HTML). HTML 使用 <> 括起来的标签来标记文档

HTML 文档类型

保存为 .html 的文档, 没有人工修改就不会改变, 被称为静态 HTML

动态 HTML, 会根据服务器应用程充或脚本程序自动生成的. 动态 HTML 需要服务器有能够自动生成 HTML 文档的网络应用程序, 这由程序设计师完成. 还要有能够将用户在网页中填写的表单发送到正确的程序, 然后 Web 服务器又能正确的把结果以 HTML 文档的方式显示给用户, 这就是公共网关接口(Common Gateway Interface, CGI)的工作

CGI 是一种基于流览器的输入, 并可在 Web 服务器上运行程序(统称为 CGI 程序)的方法. 它允许 Web 服务器与后端程序和运行脚本进行通信, 同时后端程序和脚本又可以接收用户输入信息, 最后以 HTML 页面作为用户输入响应

CGI 程序一般为脚本类语言( 如 Perl, JavaScript, VBScript 等)写的解释执行程序, 除了 CGI 程序, 现在还有在 HTML 文档中内嵌一些由编程语言写的脚本, 然后由 Web 服务器自己来执行这些脚本, 生成动态 HTML 返给客户

万维网的超文本, 超媒体, 超链接

超文本(Hypertext)

一种文本格式, 可以用超链接的方式将不同位置的文档或信息组织在一起

超媒体(Hypermedia)

是超文本的扩展, 现在一般将其等效为超文本, 因为现在 HTML 中不仅可以标记文本, 还可以标记多媒休

超链接(HyperLink)

是 HTML 最重要的属性, 是指一个网页指向目标的连接关系

HTTP 服务访问基本流程

HTTP 是面向文本的应用层协议, 默认使用的端口是 TCP 的 80 端口. 服务器有一个应用进程监听 80 端口, 当有 HTTP 报文请求时, 以 HTTP 响应报文的方式做出响应. 持续访问模式的流程

  1. 建立 TCP 连接
  2. 发送 HTTP 请求报文
  3. 返回 HTTP 响应报文
  4. 当用户关闭页面时断开 TCP 连接

HTTP 主要特征

  1. 无连接: 利用 TCP 的连接
  2. C/S 模式
  3. 高可靠性
  4. 无状态
  5. 简单快速

HTTP 请求报文格式

一个 HTPP 请求报文包括

  1. 请求行(request line)
  2. 请求头(request header)
  3. 空行(empty line)
  4. 请求主体(entity body)

请求行

包括请求方法, “URL”, “协议版本"三个字段, 最后有回车(\r)和换行(\n)控制符, 用 CRLF 表示

  • HTTP 请求方法
    请求方法 含义
    GET 请求服务器发送 URL 指定的相关 Web 页面
    HEAD 请求 URL 指定 Web 页面的相关信息而不是全部页面
    PUT 请求在服务器 URL 路径下存储一个页面, 现 GET 相反
    POST 在 URL 指定的 Web 服务器后面附加一个以 URI 格式命名的资源
    DELETE 删除 URL 字段指定的 Web 页面, 是否成功还与用户权限有关
    TRACE 指明是用来调试的环回报文
    CONNECT 用于连接代理服务器
    OPTIONS 请求查询一些特殊选项

请求头

一系列的请求头和对方兴未艾的值指出请求的附加信息以及客户端的信息. 以一系列的行表示, 行以 CRLF 结尾. 许多请求头可以有多个值, 不同的值用逗号隔开

request header 描述
Accept 指定客户端能处理的多用途互联网邮件扩展(Multipurpose Internet Mail Extension, MIME), 如 Accept:img/gif, 要求接收 gif 资源, Accept:text/html
Accept-Charset 指定客户端能接受的字符集, 如 Accept-Charset:utf-8, gb2312, gbk, 如果没有设置默认是所有
Accept-Encoding 指定客户端能解码的数据格式, 如 Accept-Encoding:gzip, compress, 默认是所有都可以
Accept-Language 指定客户端语言类型, 如 Accept-Language:zh-cn
Authorixation 指定受信任列表
Cookie 发送以前设置的 Cookie
Connections 请求特殊的连接方式, 如 Connections:Keep-Alive
Date 指定发送消息的时间
From 请求发送者电子邮箱地址
Host 指定域名和端口, 默认端口可省略
referer 包括一条 URI 信息, 要求从指定的 URI 中访问当前请求页面
User-Agent 将客户端的一些信息发送给服务器
Upgrade 客户端希望切换到新的协议

空行

提示没有请求头了 CRLF 组成的空行

请求主体

不能在 GET 中使用, 可以在 POST 方法中向服务器提供一些凭据信息.

HTTP 响应报文格式

  1. 响应行(Response)
  2. 响应头(Response Header)
  3. 空行
  4. 响应主体

响应行

协议版本, 状态码, 描述短语状态码用 3 位数表示状态

  • 100: 服务器同意处理
  • 200: 请求成功; 204:无内容, 也表示请求成功
  • 301: 页面以重定向
  • 400: 服务器无法理解请求; 401: 未经授权; 403: 服务器拒绝服务; 404: 请求页面不存在
  • 500: 服务器不可预期的错误; 503: 当前无法处理, 过会再试
