抽象工厂模式(Abstract Factory)

  1. 意图: 创建一系列相关或相互依赖对象的接口, 而无需指定具体的类
  2. 别名: Kit
  3. 动机: 有很多个类, 它们相互关联. 因为不同的要求它们有不同的系列, 将不同系列的实例化进行硬编码, 不利于维护. 定义一个抽象工厂类, 它对每个系列有一个具体的子类, 该子类的创建接口会调用相应系列类的构造器
  4. 适用性
    1. 一个系统要独立于它的产品的创建, 组合和表示时
    2. 一个系统要用多个产品系列中的一个来配置时
    3. 当你要强调一系列相关产品的设计以便联合使用时
    4. 当你提供一个产品类库, 而只想提供接口, 不想提供实现时
  5. 参与者
    • AbstractFactory
      • 声明一个创建抽象产品对象的接口
    • ConCreteFactory
      • 实现创建具体产品对象的操作
    • AbstracProduct
      • 为一类产品对象声明接口
    • ConCreteProduct
      • 定义一个会被相应工厂创建的具体产品类
      • 实现接口
    • Client
      • 仅使用抽象类提供的接口
  6. 协作
    • 通常运行时刻创建一个 ConcreteFactory 实例
  7. 效果
    • 它分离了具体的类
    • 易于更改产品系列
    • 有得于产品的一致性
    • 不过难以新的产品种类, 因为需要更改增加创建接口, 会影响抽象工厂和它的所有子类
  8. 实现
    • 一个应用中一般只需要一个工厂, 应该最好实现为一个 Singleleton
    • 创建产品: 抽象工厂为每个产品定义一个创建接口, 由它的子类来实现
    • 定义可扩展的操作: 通常需要在抽象工厂类中添加新的接口, 并修改所有相关的子类

建造者模式(Builder)

  1. 意图: 将一个复杂对象的构建与表示分离, 使得同样的构建过程可以构建不同的类
  2. 动机: 一个富文本(RTF)可以转换为不同的表示方法(HTML, ASCII, Org, Tex), 可能的表示方式是无限的, 当想要添加新的目标表示时, 不想修改复杂的文本阅读类(RTFReader), 一种方法是使用可以进行标记转换的类来配置 RTFReader(), RTFReader 只负责文档分析, 文档的转换由 TextConvert 的子类来实现
  3. 适用性:
    • 当创建复杂对象的算法应该独立于该对象的的组成部分以用它们的装配方式时
    • 当构造过程必须允许被构造的对象有不同的表示时
  4. 参与者
    • Builder: 为创建一个产品的各个部件指定抽向接口
    • ConcreteBuilder
      • 实现 Builder 的接口, 创建产品的各个部件
      • 定义并明确它所创建的表示
      • 提供一个检索产品的接口
    • Product: 表示被构造的复杂对象

工厂方法(Factory Method)

  1. 意图: 定义一个接口, 让子类决定实例化的类
  2. 别名: 虚构造器(Virtual Constructor)
  3. 动机: 用抽象类定义和管理对象间的关系
  4. 适用性
    • 当一个类不知道它所必须创建的对象的类的时候
    • 当一个类希望由它的子类来指定所应创建的对象的时候
  5. 实现
    • 定义一个创建接口, 可以是纯虚函数, 实现由子类完成. 或者提供一个默认实现
    • 定义参数化工厂, 根据参数构建对象

原型模式 (Prototype)

  1. 意图: 通过原型的实例来指定创建对象种类, 并通过拷贝这个原型来创建新的对象
  2. 动机
    1. 动行时刻增加和删除产品
    2. 改变值以指定新对象
    3. 减少工厂类的编写
  3. 适用性
    1. 当一个系统独立于它的产品创建构成和表示时
    2. 当要实例化的类是运行时指定的
    3. 为了避免创建一个与产品类层次平行的工厂类层次时
  4. 实现
    1. 原型管理器, 当原型的数目和类型不固定时, 维据一个原型注册表, 客户可以向注册表中添加和检索原型
    2. 实现克隆操用

单件模式(singleton)

  1. 意图: 保证一个类仅有一个实例
  2. 动机: 有些类只需要一个对象, 如前面所说的工厂对象. 有些类只能有一个对象, 如系统中只应有一个文件系统和一个窗口管理器.
  3. 适用性: 当类应有唯一实例且客户可访问时, 当唯一实例可通过子类扩展时
  4. 参与者: Singleton: 定义一个类方法(C++中的静态公有成员函数) Instance 来允许客户访问他的唯一实例
  5. 效果:
    • 对唯一实例的受控访问
    • 缩小名空间: 与全局变量不同, 避免了存储唯一实例的全局就是污染名空间
    • 允许对操作和表示的精化: Singleton 允许扩展子类
    • 可以控制实例的个数
  6. 实现

    • 保证唯一一个实例: 可以将类的创建操作隐藏在一个类的静态成员中, 并且可以访问保存了唯一实例的变量

    static list<NameSingletonPair>* _register;

    {
